# 技術解説 — 使用技術の詳細まとめ

このプロジェクトで使用する、一般的なアプリ開発では触れる機会の少ない専門技術をまとめる。

---

## 全体パイプラインと技術の対応

```
[Mac側]
CGVirtualDisplay          → 仮想ディスプレイ作成
ScreenCaptureKit          → 画面キャプチャ
VideoToolbox              → H.264ハードウェアエンコード
TCP Socket                → データ送信
ADB (adb forward)         → USB通信トンネル

[Android側]
TCP Socket                → データ受信
MediaCodec                → H.264ハードウェアデコード
SurfaceView               → 映像描画
```

---

## 1. ADB (Android Debug Bridge)

### 何者か
Androidデバイスを開発用PCから操作するためのコマンドラインツール。
Googleが提供するAndroid Platform Toolsに含まれる。

### このプロジェクトでの役割
Wi-Fiを使わず、USBケーブル経由でMacとAndroidの間にTCP通信トンネルを作る。

### adb forward の仕組み

```
Mac                          Android (SGP312)
localhost:9000  ←→  ADB USB  ←→  localabstract:tablet_mirror
```

```bash
adb forward tcp:9000 localabstract:tablet_mirror
```

- Mac側の `localhost:9000` に書き込んだデータが、USB経由でAndroid側の `localabstract:tablet_mirror` というUnixドメインソケットに届く
- `localabstract` = Androidのアブストラクトソケット名前空間 (ファイルを作らないUnixソケット)
- ADBデーモン (`adbd`) がUSBパケットをTCPに変換する中継役

### 注意点
- ADB接続が切れると forward 設定も消える → 再接続ロジックが必要
- `adb devices` でデバイスが `device` 状態のときのみ forward が有効

## 2. CGVirtualDisplay (macOS プライベートAPI)

### 何者か
macOSのCoreGraphicsフレームワークに含まれる非公開API。
ソフトウェアだけで仮想ディスプレイを作成し、macOSに物理モニタと同等に認識させる。

### このプロジェクトでの役割
タブレットを「Display 2」として扱うための仮想ディスプレイを作る。
これがないとウィンドウのドラッグ移動ができない。

### 仕組み

```objc
// 1. ディスプレイの仕様を定義
CGVirtualDisplayDescriptor *descriptor = [[CGVirtualDisplayDescriptor alloc] init];
descriptor.name = @"Tablet Display";
descriptor.maxPixelsWide = 1280;
descriptor.maxPixelsHigh = 800;

// 2. 仮想ディスプレイを作成 → macOSのWindowServerに登録される
CGVirtualDisplay *display = [[CGVirtualDisplay alloc] initWithDescriptor:descriptor];

// 3. 解像度モードを設定
CGVirtualDisplaySettings *settings = [[CGVirtualDisplaySettings alloc] init];
settings.modes = @[[[CGVirtualDisplayMode alloc] initWithWidth:1280 height:800 refreshRate:30]];
[display applySettings:settings];

// displayIDが発行される → これが普通のCGDirectDisplayIDとして扱える
CGDirectDisplayID displayID = display.displayID;
```

### 注意点
- `CGVirtualDisplay` オブジェクトを解放 (`display = nil`) すると即座に仮想ディスプレイが消える
- プライベートAPIのためAppleが予告なく変更・廃止する可能性がある
- App Storeへの配布には特別な権限が必要だが、個人ローカル実行は可能

## 3. ScreenCaptureKit (macOS 12.3+)

### 何者か
macOS 12.3で追加されたApple公式の画面キャプチャAPI。
特定のディスプレイ・ウィンドウ・アプリを高効率でキャプチャできる。

### このプロジェクトでの役割
仮想ディスプレイ (CGDirectDisplayID) を指定して、フレームごとに映像を取得する。

### 仕組み (旧API: CGDisplayStream)

TabletMirrorが使用している `CGDisplayStream` はScreenCaptureKitより古いAPIだが、より低レベルで直接的。

```swift
let stream = CGDisplayStream(
    dispatchQueueDisplay: displayID,   // キャプチャするディスプレイのID
    outputWidth: 1280,
    outputHeight: 800,
    pixelFormat: kCVPixelFormatType_32BGRA,  // BGRAフォーマット
    properties: nil,
    queue: .main,
    handler: { status, displayTime, frameSurface, updateRef in
        // frameSurface: IOSurface型 (GPUメモリ上のフレームデータ)
        // ここでエンコーダーに渡す
    }
)
stream.start()
```

### IOSurface とは
- GPU/CPU間で共有できるメモリ領域 (ゼロコピー)
- フレームデータをCPUにコピーせずにGPUエンコーダーに渡せる
- `CVPixelBufferCreateWithIOSurface()` でVideoToolboxが扱える `CVPixelBuffer` に変換可能

## 4. VideoToolbox — H.264ハードウェアエンコード (Mac側)

### 何者か
AppleのハードウェアエンコーダーにアクセスするためのフレームワークAPI。
Apple Silicon のMedia Engineを使い、CPU負荷ほぼゼロでH.264エンコードができる。

### このプロジェクトでの役割
キャプチャした生フレーム (BGRA/YUV) をH.264のNALユニット列に変換してTCPで送る。

### 仕組み

```swift
// 1. エンコードセッション作成
var session: VTCompressionSession!
VTCompressionSessionCreate(
    allocator: nil,
    width: 1280,
    height: 800,
    codecType: kCMVideoCodecType_H264,
    encoderSpecification: nil,
    imageBufferAttributes: nil,
    compressedDataAllocator: nil,
    outputCallback: { outputCallbackRefCon, sourceFrameRefCon, status, infoFlags, sampleBuffer in
        // エンコード完了コールバック → ここでTCP送信
    },
    refcon: nil,
    compressionSessionOut: &session
)

// 2. エンコード設定
VTSessionSetProperty(session, key: kVTCompressionPropertyKey_RealTime, value: kCFBooleanTrue)
VTSessionSetProperty(session, key: kVTCompressionPropertyKey_ProfileLevel, value: kVTProfileLevel_H264_Baseline_AutoLevel)
VTSessionSetProperty(session, key: kVTCompressionPropertyKey_AverageBitRate, value: 4_000_000 as CFNumber)
VTSessionSetProperty(session, key: kVTCompressionPropertyKey_AllowFrameReordering, value: kCFBooleanFalse) // Bフレーム無効
VTCompressionSessionPrepareToEncodeFrames(session)

// 3. フレームをエンコード
VTCompressionSessionEncodeFrame(session, imageBuffer: pixelBuffer, presentationTimeStamp: pts, duration: .invalid, frameProperties: nil, sourceFrameRefcon: nil, infoFlagsOut: nil)
```

### H.264 の構造 (NALユニット)

```
H.264ストリーム = NALユニットの連続

NALユニットの種類:
  SPS (Sequence Parameter Set) : 解像度・プロファイルなどのメタ情報
  PPS (Picture Parameter Set)  : エンコードパラメータ
  IDR フレーム                 : キーフレーム (単独でデコード可能)
  P フレーム                   : 差分フレーム (直前フレームからの差分)
  B フレーム                   : 双方向差分 → 今回は無効化 (低レイテンシのため)
```

### Annex B 形式
NALユニットをTCPで送る際のフォーマット。各NALユニットの先頭に `00 00 00 01` というスタートコードを付ける。

```
[00 00 00 01][SPS データ][00 00 00 01][PPS データ][00 00 00 01][IDRフレームデータ]...
```

## 5. MediaCodec — H.264ハードウェアデコード (Android側)

### 何者か
AndroidのハードウェアコーデックにアクセスするAPI (Android 4.1 / API 16以降)。
SGP312のSnapdragon S4 Proが持つハードウェアH.264デコーダー (`OMX.qcom.video.decoder.avc`) を使う。

### このプロジェクトでの役割
TCPで受信したH.264のNALユニット列をデコードし、SurfaceViewに直接描画する。

### 仕組み (Android 4.4 / API 19 の同期ポーリング方式)

```java
// 1. デコーダー作成・設定
MediaCodec codec = MediaCodec.createDecoderByType("video/avc");
MediaFormat format = MediaFormat.createVideoFormat("video/avc", 1280, 800);
// SurfaceHolder経由でSurfaceViewに直接描画 (CPU経由なしでGPU→ディスプレイ)
codec.configure(format, surfaceHolder.getSurface(), null, 0);
codec.start();

// 2. 入力ループ (NALユニットを供給)
int inputIndex = codec.dequeueInputBuffer(10_000); // タイムアウト10ms
if (inputIndex >= 0) {
    ByteBuffer inputBuffer = codec.getInputBuffers()[inputIndex];
    inputBuffer.clear();
    inputBuffer.put(nalUnitData);
    codec.queueInputBuffer(inputIndex, 0, nalUnitData.length, presentationTimeUs, 0);
}

// 3. 出力ループ (デコード済みフレームをSurfaceに送る)
MediaCodec.BufferInfo bufferInfo = new MediaCodec.BufferInfo();
int outputIndex = codec.dequeueOutputBuffer(bufferInfo, 10_000);
if (outputIndex >= 0) {
    codec.releaseOutputBuffer(outputIndex, true); // true = Surfaceに描画
}
```

### Android 4.4 の制限
- **非同期コールバック不可** (API 21から)。上記のような同期ポーリングループが必須
- コールドスタートに200〜400ms かかる
- ストリーム途中の解像度変更はクラッシュの原因になる

## 6. SurfaceView (Android)

### 何者か
Androidの描画コンポーネント。通常のViewと異なり、**専用のSurfaceをUIスレッドとは独立して持つ**。

### このプロジェクトでの役割
MediaCodecのデコード出力を直接受け取って表示する。

### なぜSurfaceViewか
```
通常のView:    CPU → Bitmap → View → SurfaceFlinger → ディスプレイ  (コピー多い)
SurfaceView:  MediaCodec → Surface → ディスプレイ  (ほぼゼロコピー、低レイテンシ)
```

MediaCodecの `configure()` にSurfaceを渡すと、デコード結果が直接Surfaceに書き込まれる。

## 7. TCPソケット通信

### このプロジェクトでの役割
H.264ストリームを連続送信するバイトストリーム。

### Mac側 (Swift)
```swift
// adb forward後、localhost:9000に接続
let connection = NWConnection(host: "localhost", port: 9000, using: .tcp)
connection.send(content: nalUnitData, completion: .idempotent)
```

### Android側 (Kotlin)
```kotlin
// ServerSocketでport 9000を待ち受け (localabstract経由でADBが中継)
val serverSocket = ServerSocket(9000)
val socket = serverSocket.accept()
val inputStream = socket.getInputStream()
// 連続読み込み
val buffer = ByteArray(65536)
while (running) {
    val bytesRead = inputStream.read(buffer)
    // NALユニットのパース → MediaCodecへ供給
}
```

### NALユニットのパース (受信側)
Annex B形式のストリームからNALユニットを取り出す処理。

```
受信バイト列: [00 00 00 01][データ][00 00 00 01][データ]...
                ↑スタートコードを目印に区切る
```

## 技術難易度まとめ

| 技術 | 難易度 | 難しい理由 |
|------|--------|-----------|
| CGVirtualDisplay | ★★★★☆ | プライベートAPI、ドキュメントなし |
| VideoToolbox | ★★★★☆ | コールバック非同期、SPS/PPSの取り扱い |
| MediaCodec (Android 4.4) | ★★★☆☆ | 同期ポーリング必須、古いAPIの癖 |
| H.264 Annex Bパース | ★★★☆☆ | バイナリプロトコルの理解が必要 |
| ADB forward | ★★☆☆☆ | コマンド数行だが接続管理が必要 |
| ScreenCaptureKit | ★★☆☆☆ | 公式APIだがIOSurface変換の理解が必要 |
| SurfaceView | ★★☆☆☆ | MediaCodecとのライフサイクル管理 |
| TCPソケット | ★☆☆☆☆ | 標準的 |
